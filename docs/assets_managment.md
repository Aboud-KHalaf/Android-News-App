# Assets Management in Android vs Flutter

# What Are Assets?

Assets are files that are bundled with the application and shipped to the user's device.

Common examples:

* Images
* Icons
* Fonts
* JSON files
* Audio files
* Videos

Applications use assets to display content that is not generated dynamically.

---

# Flutter Approach

Flutter has a dedicated section inside:

```text
pubspec.yaml
```

Example:

```yaml
flutter:
  assets:
    - assets/images/
```

Or:

```yaml
flutter:
  assets:
    - assets/images/logo.png
```

After running:

```bash
flutter pub get
```

Flutter automatically makes those files available.

Usage:

```dart
Image.asset("assets/images/logo.png")
```

Very simple and centralized.

---

# Android Approach

Android handles assets differently.

Instead of registering assets in a YAML file, Android uses special resource directories.

The most important one is:

```text
app/
└── src/
    └── main/
        └── res/
```

Inside `res` there are multiple folders for different resource types.

---

# The Resource System

Android treats many files as resources.

Examples:

```text
res/
├── drawable/
├── mipmap/
├── values/
├── font/
├── xml/
```

Each folder has a specific purpose.

---

# drawable

Most application images are stored here.

Example:

```text
res/
└── drawable/
    ├── logo.png
    ├── banner.jpg
    └── placeholder.webp
```

This is the folder you will use most often.

---

# Flutter Comparison

Flutter:

```text
assets/images/logo.png
```

Android:

```text
res/drawable/logo.png
```

---

# Accessing Images

In Jetpack Compose:

```kotlin
Image(
    painter = painterResource(R.drawable.logo),
    contentDescription = null
)
```

Notice that:

```kotlin
R.drawable.logo
```

is used instead of a file path.

Android generates these references automatically.

---

# What Is R?

You will constantly see:

```kotlin
R.drawable.logo
```

or

```kotlin
R.string.app_name
```

The `R` class is automatically generated by Android.

Think of it as a giant map containing all project resources.

Example:

```text
R
├── drawable
├── string
├── color
├── font
└── style
```

Every resource receives a generated identifier.

---

# Why Doesn't Android Use File Paths?

Flutter:

```dart
Image.asset("assets/images/logo.png")
```

Android:

```kotlin
R.drawable.logo
```

Advantages:

* Compile-time safety.
* Autocomplete support.
* Refactoring support.
* Fewer runtime errors.

If you rename an image, Android Studio updates references automatically.

---

# Naming Rules

Android resources have strict naming rules.

Allowed:

```text
logo.png
news_banner.png
article_image.webp
```

Not allowed:

```text
Logo.png
News Banner.png
my-image.png
```

Rules:

* Lowercase only.
* Numbers allowed.
* Use underscores.
* No spaces.
* No hyphens.

---

# mipmap

Application launcher icons are usually stored here.

Example:

```text
mipmap/
├── ic_launcher.png
├── ic_launcher_round.png
```

Referenced in:

```xml
android:icon="@mipmap/ic_launcher"
```

inside `AndroidManifest.xml`.

---

# Flutter Comparison

Equivalent to launcher icons generated by:

```bash
flutter_launcher_icons
```

---

# values

Contains XML files that define values used throughout the application.

Examples:

```text
values/
├── strings.xml
├── colors.xml
├── themes.xml
```

---

# Strings

Example:

```xml
<string name="app_name">NewsIt</string>
```

Usage:

```kotlin
stringResource(R.string.app_name)
```

---

# Flutter Comparison

Flutter often uses:

```dart
const appName = "NewsIt";
```

or localization packages.

Android has built-in resource support.

---

# Colors

Example:

```xml
<color name="primary">#FF9800</color>
```

Access:

```kotlin
colorResource(R.color.primary)
```

Although Compose projects often define colors in Kotlin files instead.

---

# Fonts

Custom fonts can be placed inside:

```text
res/font/
```

Example:

```text
font/
├── inter_regular.ttf
├── inter_bold.ttf
```

Usage:

```kotlin
FontFamily(
    Font(R.font.inter_regular)
)
```

---

# XML Resources

The folder:

```text
res/xml/
```

contains configuration files.

Examples:

```text
xml/
├── backup_rules.xml
├── data_extraction_rules.xml
```

You already saw these referenced inside:

```xml
AndroidManifest.xml
```

---

# Density-Specific Images

One of Android's most powerful features is support for multiple screen densities.

Example:

```text
drawable-mdpi/
drawable-hdpi/
drawable-xhdpi/
drawable-xxhdpi/
drawable-xxxhdpi/
```

Android automatically chooses the best image for the current device.

---

# Flutter Comparison

Flutter usually relies on:

```text
1.0x
2.0x
3.0x
```

asset folders.

The concept is very similar.

---

# Vector Drawables

Android supports vector graphics natively.

Example:

```text
res/drawable/ic_search.xml
```

Instead of storing a PNG.

Advantages:

* Smaller APK size.
* Infinite scaling.
* Better quality.

---

# Flutter Comparison

Comparable to:

```dart
flutter_svg
```

but Android supports vectors directly through the resource system.

```text
res/
├── drawable/
│   ├── placeholder_news.webp
│   ├── app_logo.png
│   └── empty_state.png
│
├── mipmap/
│   ├── ic_launcher
│   └── ic_launcher_round
│
├── values/
│   ├── strings.xml
│   ├── colors.xml
│   └── themes.xml
│
├── font/
│   └── inter_regular.ttf
│
└── xml/
    ├── backup_rules.xml
    └── data_extraction_rules.xml
```

# Assets vs Resources

A Flutter developer might think everything inside `res` is simply an asset.

Technically Android calls them:

**Resources**

because they include:

* Images
* Fonts
* Strings
* Colors
* Themes
* XML configurations

not just files.

---

# Key Takeaways

* Flutter registers assets in `pubspec.yaml`.
* Android stores most assets inside the `res` directory.
* Images usually go inside `drawable`.
* Launcher icons go inside `mipmap`.
* Fonts go inside `font`.
* Strings, colors, and themes go inside `values`.
* Resources are accessed through the generated `R` class.
* Android provides compile-time safety for resource access.
* Android resources are broader than Flutter assets because they include UI configuration and application metadata.
